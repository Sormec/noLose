CICLO DE VIDA DE SOFTWARE.

Modelo pragmático que un equipo de desarrolladores sigue para crear un producto digital.

Inestigación por Brayan Hidalgo el 20, Julio 2020

Es el proceso que se sigue para construir, entregar y hacer evolucionar el software, existen 3 etapas:

1. Planificación

En ella se sientan las bases del proyecto, se analiza el problema y se plantean las formas en las que el producto las solucionará. También se escoge el equipo de desarrollo, y posteriormente se establece el cronograma de trabajo.

2. Implementación y testeo

Es en este momento cuando todo el trabajo comienza y los programadores no paran hasta tener un prototipo del programa para su testeo. En esta fase una pequeña cantidad de usuarios llamados Beta Teasters utiliza estas "betas" o "demos" para dar sus impresiones, detectar errores y sus posibles mejoras.

3. Lanzamiento y manutención

Una vez que el programa está aprobado y con garantías se inicia la fase de lanzamiento, que básicamente es ponerlo en el mercado para que se vendan. Sin embargo, implica que el un equipo de desarrollo esté constantemente detectando fallos o realizando mejoras en el mismo, lo que conocemos constantemente como actualizaciones. Esto es sumamente necesario ya que un software desactualizado se obsoleta en menos de un año.

Los ciclos de vida más conocidos son los siguientes:

  • Cascada
  • Espiral
  • RAD(Desarrollo Rápido de Aplicación)
  • Orientado a objetos

PROCESO DE DESARROLLO DE SOFTWARE.

Un conjunto estructurado de actividades para el desarrollo eficaz de software.

Investigación por Brayan Hidalgo el 27, Julio 2020

Contiene los siguiente modelos, cada uno se adapta a un tipo de trabajo diferente:

-> Cascada

Es el más basico de todos, realizando una secuencia sin saltarse ningúna fase.

Proceso del Metodo en Cascada

-> Iterativo

Entrega un sistema completo desde el principio y se va agregando funcionalidades con cada versión, cada iteración es un ciclo de vida en cascada, pero a menor escala.

Comparación del desarrollo Iterativo con una pintura

-> Prototipado

Es efectivo cuando no se conocen los requerimientos y existe poca información con respecto a alas caracteristas que debe tener el sistema, entonces se crea un prototipo para reconocer si el sistema va en base a los requerimientos que pide el cliente.

Proceso del Metodo con Prototipado

-> Componentes

Es evolutivo por naturaleza, configura unidades, y cada una de estas es idenpendiente y funciona por si sola.

Proceso del Metodo por Componentes

-> Ágiles

Optimizar el proceso de creación del mismo, el cual era caracterizado por ser rígido y con mucha documentación. Ej: SCRUM, XP, Crystal Clear, Kanban, Splint (Metodología De Google)

Proceso del Metodo SCRUM

-> Espiral

No llevar un análisis de riesgo iterativo, por ende minimiza los riesgos, especialmente adecuado para sistemas complejos a gran escala.

Proceso del Metodo en Espiral

-> Reutilización

Software de nivel superior, creados en desarrollos anteriores aplicados a nuevos proyectos, para disminuir tiempo y costos.

Realidad del Metodo Reutilizable

-> Orientado a objetos

Cada uno de ellos introduce un proceso para el análisis y diseño de un producto o sistema, un conjunto de modelos que evoluciona fuera del proceso y una notación que posibilita al ingeniero del software crear cada modelo de una manera consistente.

Descripción grafica del Metodo Orientado a objetos

-> Métodos formales

Explota el poder de la notación y de las pruebas matemáticas.

Un metodo sumamente matemático

ESTANDARES DEL DESARROLLO DE SOFTWARE

Es un documento, establecido por consenso y aprobado por un organismo reconocido, que proporciona el uso común y repetido, reglas, pautas o características para las actividades o sus resultados, con el objetivo de lograr el grado óptimo de orden en un contexto dado.

Investigación por Brayan Hidalgo El 5, Agosto de 2020

Algunas de las organizaciónes q aprueban estos estandares son:

  • IEEE (Institute of Electrical and Electronic Engineers)
  • PMI (Project Management Institute)

Estos son los estandares más utilizados a nivel mundial:

ICE/ISO 12207

Proporcionar una estructura común para que compradores, proveedores, desarrolladores, personal de mantenimiento, operadores, gestores y técnicos involucrados en el desarrollo de software usen un lenguaje común.

Actividades de las normas ICE/ISO

SWEBOK (Software Engineering Body Of Knowledge)

Supone un paso esencial hacia el desarrollo de la profesión porque representa un amplio consenso respecto a los contenidos de la disciplina, caracteriza los contenidos, proveer una base para su desarrollo curricular y la creación de materiales de certificación.

Guia del SWEBOK

CMM (Capability Maturity Model)

Guías ampliamente aceptadas para desarrollar organizaciones de software de alto desempeño, determina el nivel de madurez del proceso de desarrollo que permita establecer un indicador de calidad del proceso.

Los 5 niveles de Madurez de CMM

CMMI (Capability Maturity Model Integration)

Ayuda a no trabajar más de lo necesario, sus técnicas ahorran tiempo y todo lo importante para poder generar más rápidamente todas estas demandas. Con base a la medición de la madurez de los proyectos.

Semejanza de un CMM con un Edificio

PMBOK (Project Management Body of Knowledge)

Una guía de buenas prácticas, el documento fue desarrollado para ser lo más universal posible para guiar adecuadamente la gestión y dirección de procesos, identificar las disciplinas, técnicas y experiencias aplicables a la dirección de proyectos y su finalidad de este conjunto es minimizar el riesgo de que un proyecto no alcance sus objetivos.
